Franz Panel - Improvement Plan

Layout, UX, data integrity, and plumbing alignment audit for panel.html

Layout and Resize

#1 Only horizontal split exists -- no vertical split between swarm and event log layout bug

The current layout has one draggable gutter splitting the screen left/right. The right pane has a fixed max-height:120px event log at the bottom. There is no way to resize the swarm rows area vs the event log area. The event log is hardcoded to 120px regardless of screen height, wasting space on tall screens and being too small on short ones.

The request is for a single intersection point that controls all three quadrants simultaneously. This requires a CSS grid with both a column gutter and a row gutter on the right pane, meeting at one draggable corner handle.

Proposed structure:

body
  #root  (CSS grid: [left-col] [v-gutter 5px] [right-col] / [top-row] [h-gutter 5px] [bottom-row])
    #pane-canvas      grid-column:1 / grid-row:1/4  (spans full height, left side)
    #v-gutter         grid-column:2 / grid-row:1/4  (vertical divider, full height)
    #pane-swarm       grid-column:3 / grid-row:1    (top-right: agent rows)
    #h-gutter         grid-column:3 / grid-row:2    (horizontal divider, right side only)
    #pane-log         grid-column:3 / grid-row:3    (bottom-right: event log)
    #corner-handle    positioned at intersection of v-gutter and h-gutter

The corner handle sits at the intersection. Dragging it updates both splitX (column) and splitY (row) simultaneously. Each quadrant then fills its grid cell completely. The canvas pane spans all rows on the left.

#2 --split-x CSS variable in :root is dead -- JS overwrites it before first paint dead CSS

:root declares --split-x:55% and #root uses var(--split-x) in grid-template-columns. JS calls applyLayout() synchronously on load which sets rootEl.style.gridTemplateColumns as an inline style, overriding the CSS variable before any paint. The variable is never rendered. Remove it and set the initial grid via JS only, consistent with how resize works.

// remove from :root:
--split-x: 55%

// remove from #root rule:
grid-template-columns: var(--split-x) 5px 1fr

// JS already handles this correctly via applyLayout() on load
#3 fitCanvas is not called when the right pane resizes vertically layout bug

fitCanvas() is called on window resize and on horizontal gutter drag. If a vertical gutter is added (issue #1), dragging it changes the canvas pane height but fitCanvas() is not triggered. The canvas will not re-center or rescale until the next window resize event. The vertical gutter drag handler must also call fitCanvas().

#4 canvas-stack uses fixed px sizing set by JS but canvasWrap uses flex centering -- mismatch on resize layout

canvasStack is a position:relative div whose width/height are set in px by fitCanvas(). The two canvas elements inside it are position:absolute. This works but the stack div itself has no intrinsic size before the first frame arrives, causing canvasWrap to have nothing to center. A min-width/min-height of 1px on canvasStack prevents a zero-size layout flash. Also canvasStack should not use position:relative with absolute children for this use case -- both canvases should be stacked via CSS grid (grid-area:1/1) which is cleaner and does not require the absolute positioning hack.

/* cleaner stack without absolute positioning */
#canvas-stack { display: grid }
#canvas-stack canvas { grid-area: 1 / 1 }
#c-overlay { pointer-events: none }

Swarm Panel UX

#5 swarm-rows does not auto-scroll to the most recently updated agent UX

When a new swarm message arrives, upsertAgentRow updates the existing row in place but does not scroll it into view. The most active agent (Grok, which receives the most messages) may be off-screen. The row that was just updated should be scrolled into view unless the user has manually scrolled away, in which case auto-scroll should be suppressed to avoid fighting the user.

Implementation: track whether the user has scrolled up manually. If swarmRows.scrollTop + swarmRows.clientHeight >= swarmRows.scrollHeight - 40 (near bottom), auto-scroll the updated row into view. If the user has scrolled up, do not auto-scroll.

// after upsertAgentRow updates a row:
const nearBottom = swarmRows.scrollHeight - swarmRows.scrollTop - swarmRows.clientHeight < 60;
if (nearBottom) agentRows[agent].row.scrollIntoView({block: 'nearest', behavior: 'smooth'});
#6 agent-row-body max-height:400px is a hardcoded guess that clips long outputs UX

The expanded body uses max-height:400px for the CSS transition. A 2B model output can easily exceed 400px of rendered text. The content is clipped with no scroll and no visual indicator that it is clipped. The body should be scrollable when expanded, or max-height should be set to a percentage of the pane height rather than a fixed pixel value. Given the new three-quadrant layout, max-height:calc(var(--swarm-h) - 120px) or simply overflow-y:auto on the body with a large max-height would work.

.agent-row-body.open {
    max-height: 60vh;   /* relative to viewport, not hardcoded */
    overflow-y: auto;
    padding: 6px 12px;
}
#7 agent-row-preview uses -webkit-line-clamp but the row is updated in place -- stale preview on collapse UX

The preview div shows 2 lines of the last message. When the row is open and a new message arrives, preview.textContent is updated but the user sees the expanded body. When they collapse, the preview now shows the new message correctly. This is fine. However the preview and the body text are set to the same value -- the full text. The preview is just a CSS-clamped version of the full text. This is correct and no change is needed here, but it is worth noting that the preview div is redundant with the body textEl -- they always contain identical content. The preview could be removed and the body could show a 2-line clamp when collapsed, expanding on click. This removes one DOM element per agent row.

/* collapsed state: clamp the body text directly */
.agent-row-body { -webkit-line-clamp: 2; display: -webkit-box; -webkit-box-orient: vertical; overflow: hidden }
.agent-row-body.open { -webkit-line-clamp: unset; display: block; overflow-y: auto }
/* removes the need for agent-row-preview entirely */
#8 event-log prepends entries (newest at top) but swarm-rows appends (newest at bottom) -- inconsistent direction UX

uiLog() uses eventLogEl.prepend() so the newest log entry is at the top. The swarm rows are appended to swarmRows so the newest agent row is at the bottom. Since rows are upserted in place (one row per agent, never appended), the order is fixed at first-seen order. This is fine. But the event log scrolling direction is opposite to what a user expects if they think of both panels as "latest at bottom". The event log should also append and auto-scroll to bottom, consistent with the swarm panel direction.

Status Bar and Counters

#9 "cycle: N" in the status bar shows swarmTotal which is the raw swarm message index -- not a cycle count misleading data

The status bar shows "cycle: N" where N is swarmTotal -- the total number of swarm wire messages ever received. This is not a cycle count. One brain cycle produces approximately 16 swarm messages (5 agents x 3 debate rounds + Grok input + Grok output). Showing the raw message index as "cycle" is misleading. Options:

1. Remove the counter entirely -- it adds no actionable information.

2. Rename it to "msgs: N" to accurately describe what it counts.

3. Remove it and replace the status bar with only: brand | seq | SSE status. That is all the information that is actually useful at a glance.

The "seq" counter (frame sequence number) is useful -- it tells you how many frames have been captured and annotated. Keep that. Remove "cycle".

/* proposed minimal status bar */
Franz  |  seq: 42  |  (SSE dot) SSE
#10 badge-img shows "seq N ok/fail" -- redundant with sb-seq and canvas-status redundant info

The canvas pane header shows a badge with "seq 42 ok" or "seq 42 fail". The status bar already shows the frame seq number. The canvas-status overlay already shows the canvas resolution. The badge duplicates seq from the status bar and adds ok/fail which is only relevant for ~200ms during annotation. After annotation completes it just shows a stale "ok" badge that means nothing. Replace the badge with a simple annotation status indicator: a colored dot that pulses during annotation and goes green/red on completion, then fades. Or remove it entirely and rely on canvas-status for resolution info.

Data Integrity and Plumbing Alignment

#11 Panel modifies no hub data -- plumbing contract is clean with one exception mostly clean

The panel is correctly passive. It receives data via SSE and GET /frame, renders it, and POSTs back only the annotated image via /annotated. It does not modify swarm messages, agent states, config, or event log entries. The only POST it makes is /annotated which is a required part of the annotation pipeline -- the hub cannot function without it. This is correct architecture and requires no change.

The panel does not need any hub plumbing changes to implement all improvements in this plan.

#12 fallbackPoll starts immediately on load before SSE has a chance to connect plumbing

The last two lines of the script are:

fallbackTimer = setInterval(fallbackPoll, POLL_MS);
startSSE();

fallbackPoll starts running immediately, before SSE has connected. The first poll fires at 2500ms. If SSE connects within that window (which it always will on localhost), the poll fires once unnecessarily and then gets cleared. This is harmless but contradicts the stated rule that fallback polling only activates after SSE failure. The fix is to not start fallbackTimer at all on load -- only start it inside es.onerror. startSSE() alone is sufficient on load.

// remove these two lines:
fallbackTimer = setInterval(fallbackPoll, POLL_MS);  // remove
startSSE();  // keep

// fallbackPoll only starts inside es.onerror, which is already the case
#13 SSE reconnect is never attempted after connection loss plumbing

When SSE errors, the panel switches to fallback polling and never tries to reconnect SSE. EventSource has built-in reconnect behavior but only if the connection was cleanly closed by the server. If the hub restarts, the EventSource object is in an error state and will not reconnect automatically in all cases. The panel should attempt to recreate the EventSource after a delay when SSE is lost, so it can recover when the hub comes back up without requiring a page reload.

es.onerror = () => {
    if (sseConnected) {
        sseConnected = false;
        sseDot.className = 'sse-dot dead';
        if (!fallbackTimer) fallbackTimer = setInterval(fallbackPoll, POLL_MS);
    }
    // attempt SSE reconnect after 5s
    setTimeout(startSSE, 5000);
};
#14 handleFrame has no guard against being called with a seq older than the last completed annotation plumbing

handleFrame guards with seqNum <= lastPendingSeq. lastPendingSeq is set to seqNum at the start of handleFrame and never reset on failure. If handleFrame fails mid-way (fetch error, postAnnotated returns false), lastPendingSeq remains at the failed seq. The next SSE frame event with the same seq (hub retries are not implemented but fallbackPoll could trigger it) would be silently dropped. This is acceptable given the current hub does not retry, but the variable name lastPendingSeq is misleading -- it is actually lastAttemptedSeq. Rename for clarity.

#15 cycleCount variable declared but never used dead code

let cycleCount = 0 is declared at module scope and never read or written after declaration. Remove it.

let cycleCount = 0;  // remove entirely
#16 --radius CSS variable declared but never referenced dead CSS

--radius: 10px is in :root but no CSS rule uses var(--radius). Remove it.

#17 OffscreenCanvas merge block duplicated in exportAnnotated and btn-save-log handler duplicate code

The save handler duplicates the OffscreenCanvas draw+export logic instead of calling exportAnnotated(). The save handler should call exportAnnotated() directly and convert the returned base64 to a data URI for the img tag.

// btn-save-log handler -- replace the duplicate block with:
const annB64 = await exportAnnotated();
const canvasDataUri = annB64 ? 'data:image/png;base64,' + annB64 : '';
#18 document.getElementById('badge-img') called 3x uncached inside handleFrame duplicate query

badge-img is queried three times inside handleFrame on every frame. Cache it at module init alongside the other element references.

const badgeImg = document.getElementById('badge-img');
const sbSeq = document.getElementById('sb-seq');
const sbCycle = document.getElementById('sb-cycle');
// then use badgeImg directly in handleFrame and handleSwarmMsg

Hub Plumbing Changes Required

None of the improvements above require any changes to franz_hub.py or brain_agentic.py. The panel receives all necessary data through existing SSE events (state, frame, swarm, agent_status, log) and the existing GET /frame + POST /annotated pipeline. The three-quadrant layout, auto-scroll, SSE reconnect, dead code removal, and counter cleanup are all purely panel-side changes. The hub plumbing contract is already correct for what the panel needs.
