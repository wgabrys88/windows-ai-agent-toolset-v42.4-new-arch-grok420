Franz - Duplication and Dead Code Analysis

Cross-file audit: exact duplicates, structural duplicates, dead code, and merge candidates

Exact Duplicates

#1 NORM = 1000 defined in three files independently 3 files

The constant NORM = 1000 is declared at module level in franz_hub.py, win32.py, and as const NORM = 1000 in panel.html. All three refer to the same coordinate space contract. There is no single source of truth.

franz_hub.py:  NORM: int = 1000
win32.py:      NORM: int = 1000
panel.html:    const NORM = 1000;
#2 _clamp / _clamp_norm are identical functions in two files franz_hub.py + win32.py

_clamp in franz_hub.py and _clamp_norm in win32.py are logically identical. Both do max(0, min(NORM, value)). They exist separately because win32.py is a subprocess and cannot import from franz_hub.py, but they are a maintenance risk if NORM ever changes.

franz_hub.py:  def _clamp(value: int) -> int:
                   return max(0, min(NORM, value))

win32.py:      def _clamp_norm(value: int) -> int:
                   return max(0, min(NORM, value))
#3 Action type enumeration repeated four times across the codebase brain + hub + win32 + panel

The full set of action names (click, double_click, right_click, type_text, press_key, hotkey, scroll_up, scroll_down, drag, wait) is enumerated in four separate places. Any new action type requires changes in all four.

brain_agentic.py  _CAPTAIN_SYSTEM:   lists all 10 action types in JSON example
brain_agentic.py  _dispatch_action:  match act: case "click" ... case "drag"
franz_hub.py      _execute_action:   match action_type: case "click" ... case "drag"
win32.py          main():            match command: case "click" ... case "drag"

#6 VLM call body duplicated between call_vlm_orchestrator and call_vlm_agent franz_hub.py x2

Both functions contain the same pattern: set status, acquire semaphore, sleep delay, call _do_vlm_call, set status idle, return result. The only structural difference is which semaphore is acquired and the pre-acquire status label. The shared body is ~8 lines duplicated verbatim.

call_vlm_orchestrator:
    set_agent_status(agent_name, "thinking")
    async with _vlm_orchestrator_semaphore:
        if delay > 0: await asyncio.sleep(delay)
        result = await _do_vlm_call(...)
        set_agent_status(agent_name, "idle")
        return result

call_vlm_agent:
    set_agent_status(agent_name, "awaiting_vlm")
    async with _vlm_agent_semaphore:
        set_agent_status(agent_name, "thinking")
        if delay > 0: await asyncio.sleep(delay)
        result = await _do_vlm_call(...)
        set_agent_status(agent_name, "idle")
        return result

Merge Candidates

#17 _dispatch_action in brain and _execute_action in hub are two halves of one pipeline brain + hub

_dispatch_action in brain_agentic.py translates Grok JSON fields into hub action dicts. _execute_action in franz_hub.py translates hub action dicts into win32.py subprocess commands. Both are match blocks over the same 9 action types. The brain layer adds zero value here -- it is pure mechanical translation from one dict schema to another. The hub could accept the Grok JSON fields directly, eliminating _dispatch_action entirely and one full layer of translation.

Grok JSON -> _dispatch_action (brain) -> hub action dict -> _execute_action (hub) -> win32 args
                    ^--- this layer is pure schema translation, no intelligence
#18 call_vlm_orchestrator and call_vlm_agent can be unified with a semaphore parameter franz_hub.py

The two VLM call functions are structurally identical except for which semaphore they acquire and whether they set awaiting_vlm before acquiring. A single internal _call_vlm(semaphore, pre_status, messages, ...) would replace both, with the two public functions becoming one-line wrappers. This removes ~20 lines of duplicated logic.

async def _call_vlm(sem, pre_status, messages, temperature, max_tokens, agent_name):
    if agent_name and pre_status: set_agent_status(agent_name, pre_status)
    async with sem:
        if agent_name: set_agent_status(agent_name, "thinking")
        if delay > 0: await asyncio.sleep(delay)
        result = await _do_vlm_call(messages, temperature, max_tokens)
        if agent_name: set_agent_status(agent_name, "idle")
        return result
#19 _do_click, _do_double_click, _do_right_click share extractable preamble win32.py

All three click functions share the same 4-line preamble: parse pos, resolve screen pos, move cursor, settle delay. Extracting a _move_to(pos_str, region_str) helper that returns nothing and performs the move+settle would remove 12 lines of duplication and make the intent of each function clearer.

def _move_to(pos_str: str, region_str: str) -> None:
    nx, ny = _parse_pos(pos_str)
    px, py = _resolve_screen_pos(nx, ny, region_str)
    _move_cursor(px, py)
    time.sleep(CONFIG.click_settle_delay)

def _do_click(pos_str, region_str):
    _move_to(pos_str, region_str)
    _mouse_event(LEFT_DOWN); time.sleep(...); _mouse_event(LEFT_UP)

#21 SSE state and agent_status events carry overlapping agent data to the panel franz_hub.py + panel.html

The hub publishes a full agents dict inside every state event AND a separate agent_status event on every individual status change. The panel handles both. The state event already contains all agent statuses, making agent_status redundant for the panel's chip display. Either the panel should rely only on agent_status events (removing the agents loop from the state handler), or state should drop the agents field and the panel should rely solely on agent_status. Currently both paths run and both call upsertChip, doing the same DOM update twice per status change.
